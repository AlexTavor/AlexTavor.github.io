dod

A dashboard of dashboards

dod is one local supervisor that registers, launches, supervises, and frames every other local dashboard in a single admin UI. It is a process supervisor with a real interface, not a launcher menu. Owning each dashboard's lifecycle is what ends the orphaned-process and stale-port mess that a folder of half-running dev servers turns into.

The model

The left list is the catalog and the spawn surface at once: click an entry to open it, starting it if it was stopped, with inline stop, restart, and forget. There is no header tab-strip and no separate add step. Native dashkit dashboards render in place from their spec; everything else is shown in a frame, and is demoted to an open-in-a-new-tab link if it refuses to be framed.

A child process is killed when dod exits, and dod itself is kept alive by launchd. On boot it re-adopts any survivors into the kill set, so the single-gate property holds even across an unclean crash. dod lives in ~/.dod and supervises dashboards across many repos, so one instance covers every project rather than one dod per project.

Providers

A provider turns something in the world into dashboard entries dynamically. The bundled PDD provider scans for .pdd/ repos and emits a findings dashboard and a plan dashboard per repo, with dod assigning each a stable unique port. That is why Proof-Driven Development runs as a provider inside the one dod, rather than a separate dashboard server per project.

Why it replaced its predecessor

dod replaces an earlier single 1075-line script that worked on the happy path but accreted residue: an append-only registry with no way to forget an entry, in-memory ownership that evaporated on restart, liveness that under-reported death, and stop and restart that silently did nothing. dod keeps the parts that were right (process-group tree-kill, contract-gated discovery, a same-origin spec proxy) and fixes the rest, as a tested package.
